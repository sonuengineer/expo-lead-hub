# Deploy — Booth Setup (Free Tier)

**Stack:** Supabase (DB, done) · Render free (backend) · Vercel (frontend).

Everything the app calls stays same-origin: the frontend uses relative `/api`, and
`vercel.json` proxies `/api/*` to the Render backend — so there's no CORS to configure.

---

## 0. Push the repo to GitHub (one time)
Render and Vercel deploy from a Git repo.

```bash
git init
git add .
git commit -m "Exhibition Lead Capture"
# create an empty repo on github.com, then:
git remote add origin https://github.com/<you>/<repo>.git
git branch -M main
git push -u origin main
```
> ⚠️ Confirm `.env` files are **git-ignored** (`git status` must NOT list `apps/backend/.env`
> or `packages/db/.env`). Secrets go into the Render/Vercel dashboards, never into git.

---

## 1. Backend → Render
1. Render dashboard → **New → Blueprint** → select this repo. It reads `render.yaml`.
2. When prompted, set the secret env vars (copy the two URLs from your local
   `apps/backend/.env`):
   - `DATABASE_URL` = the Supabase **pooler** URL (`…:6543/…pgbouncer=true&connect_timeout=15&schema=elc`)
   - `DIRECT_URL`   = the Supabase **session** URL (`…:5432/…connect_timeout=15&schema=elc`)
   - `CORS_ORIGIN`  = leave blank for now (fill after step 2).
   - `JWT_SECRET` is auto-generated by Render.
3. Deploy. When it's live, copy the URL, e.g. `https://elc-backend.onrender.com`.
4. Test: open `https://elc-backend.onrender.com/api/health` → `{"status":"ok"}`.

> The DB is already migrated & seeded (the `elc` schema), so no migration runs on Render.
> If you ever need to re-seed, do it locally with `pnpm db:seed` — it targets Supabase.

---

## 2. Point the frontend at the backend
Edit **`apps/frontend/vercel.json`** — replace `YOUR-BACKEND.onrender.com` with your real
Render host. Commit & push:
```bash
git commit -am "point frontend at render backend" && git push
```

---

## 3. Frontend → Vercel
1. Vercel → **Add New → Project** → import the repo.
2. Set **Root Directory = `apps/frontend`** (Framework auto-detects **Vite**).
3. No env vars needed (the `vercel.json` rewrite handles the API).
4. Deploy. Copy the URL, e.g. `https://elc.vercel.app`.

---

## 4. Close the loop
- In **Render** → the backend service → Environment → set `CORS_ORIGIN` = your Vercel URL
  (`https://elc.vercel.app`) → save (it redeploys).
- Open the Vercel URL → log in `admin@elc.com` / `admin123` → **change the admin password**
  (create a new admin / rotate — don't run a live booth on the default creds).

---

## 5. Keep the free backend awake (avoid cold starts)
The free Render service sleeps after ~15 min idle (first hit after that waits ~40s).
Set a free pinger: **cron-job.org** → new cron job → URL `https://elc-backend.onrender.com/api/health`
→ every 10 minutes. Keeps it warm during booth hours.

---

## 6. Use it at the booth
- **QR / visitor form:** admin (on the Vercel URL) → **QR Codes** → generate per booth ×
  visitor type → print. Scanning opens `https://elc.vercel.app/v/<code>` → the public form.
- **Card scan:** staff logs in on a phone (Vercel URL is HTTPS, so the camera works) →
  **Scan Card** → photo → review → submit. The lead records who scanned it.
- Watch results live in **Dashboard / Leads**.

---

## Caveats (free tier)
- **Cold start** on the free backend — the pinger (step 5) mitigates it; for a busy event,
  upgrade the Render service to a paid instance (~$7/mo) for always-on + more RAM.
- **OCR** (Tesseract) is memory-heavy; on 512 MB it can be slow or occasionally fail. QR
  forms are light and unaffected. For fast OCR, add a Google Vision key later.
- **No offline mode yet** — if venue Wi-Fi drops, submissions fail (Phase 12 not built).
